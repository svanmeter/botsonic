Introduction

Databases managed by the SQL Server Express database engines have a documented size limit of 4GB for SQL 2005 and 10GB for SQL 2008 and later.  Whilst this is adequate for most Syrinx users to store many years’ worth of information, it is likely that at some point in the future this limit may be reached.  An option exists within the Syrinx Database Administration program that allows a database to be compacted, and optionally allows old data to be removed from the database.

Compact Database option

This option is on the main menu of the Syrinx Database Administration program:

As with point release upgrades, all users must be logged off of the system before running the utility.  When the user presses OK, the following steps are taken:

 

Index rebuild

The first step is to rebuild all indexes for all tables in the database.  By re-indexing the tables, any fragmentation in the indexes is removed.  This is done by issuing DBCC DBREINDEX commands.  This is recommended in Microsoft documentation over dropping and re-creating the indexes: “Because the work is done by one statement, DBCC DBREINDEX is automatically atomic, while individual DROP INDEX and CREATE INDEX statements would have to be put in a transaction to be atomic. Also, DBCC DBREINDEX can take advantage of more optimizations with DBCC DBREINDEX than it can with individual DROP INDEX and CREATE INDEX statements.”

 

Shrink database

The following command is then executed to shrink the database - DBCC SHRINKDATABASE(SyrinxDM, 10).  This relocates rows at the end of the database file to unallocated pages within the file, and then truncates the file and releases the space freed back to the operating system.  Note: The target size for the data file as calculated by DBCC SHRINKDATABASE can never be smaller than the size specified when the file was originally created.

 

Truncates the transaction log

The following command is executed - BACKUP LOG SyrinxDM WITH TRUNCATE_ONLY.  From Microsoft documentation: “This removes the inactive part of the log without making a backup copy of it and truncates the log. This option frees space. Specifying a backup device is unnecessary because the log backup is not saved after backing up the log using either NO_LOG or TRUNCATE_ONLY, the changes recorded in the log are not recoverable. For recovery purposes, immediately execute BACKUP DATABASE.”

 

Backup database

As recommended by Microsoft, a database backup is then taken, and the backup saved in the file SQLDataSp\Backup\SyrinxDM_Post_Compact_DDMMMYY_HHNNSS.bak.
 

Data Removal

The compaction described will free up space taken by natural fragmentation that occurs during normal operation, but will not remove any data from the database.  As an additional option, you can also choose to remove old data from the database during the compaction process.  To remove old data, press Ctrl+F5 after opening the ‘Compact Database’ screen:

Note: In order to remove data that is aged less than the default values of 4 and 7 years, a patch must be run against the Syrinx database prior to starting the data removal process.

 

 Upon pressing OK, old data from the following areas is deleted:

 
Temporary tables

All data from the temporary tables is deleted.

 
Purchase

Data is removed from purchase receipts, purchase orders and purchase invoices. Purchase receipts are removed when they are older than four years along with their lines, stock movements and candidate stock items.  Purchase orders are removed when they are older than four years along with their lines and change log.  Purchase invoices are removed when they are older then seven years along with their lines and reconciliation with purchase receipts.

 
Advice notes

Data is removed when older than four or seven years dependent on the type of data. Logs and stock movements are removed when they are older than four years, whereas cash transactions, lines and notes themselves are removed when they are more than seven years old.  Additionally, advice notes which remain in the system and have had some data removed (like stock movements or logs) are marked with an appropriate flag, so they are not visible within the system.

 
Contracts

Contracts, their items and appropriate cash transactions are removed when they are older than seven years old. Other data linked with contracts (like logs, operators, timesheets, missing timesheets confirmations, charge units, purchase orders, exchange notes with items, offhire notes with items, service notes with items, stock and site movements, days not on hire, intersection with between invoices and history of previews sites) is removed when it is older than four years.  Additionally, contracts which remain in the system and have had removed some data are marked with a appropriate flag, so they are not visible within the system.

 
Invoices

All invoices and all linked data (like lines and intersections with contracts) are removed after seven years. 

 
Customers

Customer are removed from the system if they have been created earlier than seven years ago and are not linked with advice notes, hire contracts, service jobs or hire invoices. 

Together with a customer, linked contacts, addresses, pricing agreements, mail lists, invoicing lists, correspondence and intersection with depots are also removed.  

 
Suppliers

Suppliers are removed from the system if they have been created earlier than seven years ago and are not linked with any purchase orders, purchase invoices, purchase receipts, hire contracts, service jobs or exchange notes. 

While removing a supplier, other linked data like contacts, intersection with depots, mail lists, addresses and supplier stock items is also removed.

 
Stock items

Stock items are removed when both their date of creation and disposal date are earlier than seven years and there is no quantity in stock, as long as the item is not referenced by any advice note lines, hire items, purchase receipt lines, purchase order lines, service job lines, stock assignments, period end levels, depot transfer lines and stock build components. 

When a stock item is removed, other linked data (like item logs, stock service reminders, candidate stock items, stock levels, stock movements, documents, equipment inclusions,  transport rates, supplier stock items, part lists usage, parts list entries, unit price in different currency and intersection with contract items) is also removed. Together with stock items, data from stock movements and depot transfers is removed if it is older than four years and from period ends if it is older than seven years. Additionally data is removed from stock checks and build notes. Stock checks (like stock check items, individual number items and stock checks) are removed when they are older than four years. Build notes (like build item lines, stock movements and build notes) are removed when they are older than seven years. Customer owned stock items are removed if there have not been any service jobs within last seven years.

 
Sales and Hire categories

Categories are removed when they do not have any reference to stock items, stock build components, stock part list entries, hire items, advice note lines, build item lines, supplier stock items, purchase order lines, purchase receipt lines, service job lines, depot transfer lines, individual hire rates discounts and individual sales rates discounts.

When categories are removed, linked data like documents, intersection with items or with another category, standard rates for equipment, intersection with reminders types and stock parts list usage is also removed .

 
Operational Data

Operational data is removed from many different tables.

During this operation data from till balances, bank holidays and system events is removed when they are older than seven years, while data from system error log, system logons and logs for transferring data into account systems is removed when it is older than four years. Data is also removed from invoice batches, invoice batch prints, import entries and invoice delivery exports.
