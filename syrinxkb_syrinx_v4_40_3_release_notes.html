Syrinx Client Support for GMAIL OAuth 2 Sign-in – On May 30, 2022 google will be disabling the option to allow applications and physical devices to connect to Gmail simply by using the account’s username/password.  Syrinx now has the option to use more secure sign-in methods that will allow apps to continue to link with GMail accounts in order to accommodate this new requirement.  This functionality is turned on for the Syrinx client by enabling the new system option “Use Gmail OAuth2?” on the “Email” tab of Syrinx SP Configuration\System Options\General:

As with the previous CDO email settings, users wishing to use Gmail OAuth2 for emailing will be required to have their GMail account email address set against their Syrinx logon in Syrinx SP Configuration.  In most instances the user will already have their email address set against their Syrinx user record if they have already been emailing from the system.

In the main Syrinx program, users can then indicate that they are using the new email option by setting the “Send Using” value on the “Email” tab of Syrinx SP\Users\User Setup to the new “GMail” option:

Selecting this value will enable a new “Reg. with Gmail” button and disable the other SMTP fields as this information is no longer required.  The user will then need to register with GMail before they can email from the Syrinx client. 

Pressing the “Reg. with Gmail” button will open the Google authentication page and the user must login using their GMail credentials.  The user will then be presented with a screen detailing what levels of access to allow the application. At minimum the apps require the “Send email on your behalf” permission. Press continue after enabling the required option:

In response the user will see their email address being echoed back. This indicates that the email account is now registered and is ready for use and the GMail sign-in page can be closed.  Once a specific email address has been registered it will not need to be re-registered if it has been linked with other Syrinx Logons but other users who are linked to this email address will still need to set the “Send Using” value in Syrinx clients to the new “GMail” option.  Registering an email address should be a one-time process but can be repeated should the app lose permission for some reason. The behaviour when sending emails from the Syrinx client is unchanged.

If a user also emails invoices from the Syrinx client they will also need to set the “Send Emails” value on the “Batch Invoice PDFs” tab of the “User Options” screen to a the new “GMail – Send Immediately” option: 

Syrinx Add-On Module Support for GMAIL OAuth 2.0 Sign-in – The syrinx add-on modules that have email capabilities have also been updated to work with the new GMail security requirements.  Unless mentioned otherwise below, these add-ons will use the email address linked to the Syrinx logon that is used to establish connection to the Syrinx database.  As these might not be actual “User” accounts, a new “Reg. with GMail” button has been added to the “Amend Logon User” screen in Syrinx SP Configuration so that these emails can be registered without having to log into the main Syrinx program:

This button will only be visible against a Syrinx logon record if the new system option “Use Gmail OAuth2?” has been enabled on the “Email” tab of Syrinx SP Configuration\System Options\General.  Pressing this button will launch the Google sign-in page detailed previously in these release notes.  

The following setting/changes will then be required in the specific Add-on modules themselves in order to enable the new functionality:

Task Scheduler – A new “Use Gmail OAuth2?” option has been added to the scheduler’s configuration screen.  If this option is enabled, the email linked to the Syrinx logon set against the “Logon Name:” field must be registered with GMail.  The “Send From:” value must also match that email address:

If the service is running when this option is enabled, the service should be restarted to pick up this configuration change.

Mobile Service for Mobile Apps– A new key value, “UseGMail” has been added to the site’s smtp.config file.  When this is set to “true”, the mobile service will use the new methods when connecting to the GMail account.  The email address set against the Syrinx Logon set in the site’s Syrinx.config file must be registered with GMail:

If the site is running when the smtp.config file is updated, its application pool should be restarted to pick up this configuration change.

Transfer Tool – A “Use GMail OAuth2” option has been added to the “Run as Service” tab on the transfer tool configuration settings screen.  If this option is enabled, the email linked to the Syrinx logon set on the “Syrinx Data” tab of the configuration screen must be registered with GMail.  

If the service is running when this option is enabled, the service should be restarted to pick up this configuration change.

SyrinxWS – A “Use GMail OAuth2” option has been added to the “SMTP” tab of the SyrinxCWA site’s configuration page.  Enabling this option will set all SyrinxWS sites (e.g. CWS, CWO, CWA) to use the new method of connecting to GMail.  The email address set against the Syrinx Logon set in the sites’ Syrinx.config files must be registered with GMail:

All of the “Send From” values on the other tabs of the configuration page should be set to the email linked to the Syrinx logon.  If the sites are running when this option is enabled, the SyrinxWS application pool should be restarted to pick up this CWA configuration change in the other sites.

Syrinx SalesApp – No special configuration is required for the Syrinx SalesApp as it will simply observe the “Use Gmail OAuth2?” setting on the “Email” tab of Syrinx SP Configuration\System Options\General.  If this option is set, the site will try to use the email address linked to the Syrinx user logged into the SalesApp as long as that email address has been registered.  If the site is running when the “Use Gmail OAuth2?” setting is enabled, the SalesApp application pool should be restarted to pick up this change.

*Note – the SalesApp only emails error messages to the email address set as the “NotifyEmailAddress” value in the site’s application settings.  

Purchase Order Search in Syrinx365 – A search grid for consumable purchase orders has been added to Syrinx365 allowing users to search for and filter on purchase order records.  As with previous search grids, the users can save their search preferences for later use.

Allow Description Value to Wrap on Mobile Printed Document - The description text (e.g. the "Desc:" value) on the mobile job document printed from the Workshop App will now wrap to multiple lines in order to display the full text for this field.

Work Type Codes Now Mandatory When Confirming Hire Contracts - If a customer record has current "Work Type Codes" set against it on the "Work Codes" tab of their customer record, users will not be able to print or confirm contracts for that customer without linking the contract to one of the customer's available work type codes and will be issued with the following prompt:

"You have omitted to enter a value for Work Type Code.  Please enter one now"

If the contract is linked to a non-current work code, the user will be advised that the specific work type code is no longer current and to select a new one.  The non-current work type code could also be made current again in the Syrinx SP Configuration program to allow the contract to be confirmed if it is indeed valid for that hire.

Customer/Supplier Records Can Only be Linked to Appropriate Tax Codes - The tax code dropdown fields on customer and supplier records will now only display tax codes that are applicable to that type of record meaning that only tax codes that have been enabled for sales will be available to set as defaults on customers and only tax codes that have been enabled for purchase will be available to set as defaults on suppliers.

Promised Date and Time columns Added to Unscheduled Grid in Driver Scheduler - Promised Date and Time columns have been added to the lower grid of the driver scheduler that displays unscheduled job information.  Users can filter and sort on these new columns.  Previously, this information was only visible in the job details pane after clicking on a specific job.
