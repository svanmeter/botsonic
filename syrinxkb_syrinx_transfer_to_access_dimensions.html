Overview
The transfer program to Access Dimensions provides a two way link between the Syrinx and Access Dimensions databases. Transfer of Customers, Suppliers, Sales & Purchase Invoices and Cash Payments is supported, as is the transfer back of customer current/aged balances & unpaid invoices, plus modified customer/supplier names and addresses.  

Configuring Syrinx for use with Access Accounts

You will need to set up tax codes for use with Access dimensions in Syrinx.  Generally, Access Dimensions will use “0” for zero rate and “1” for standard rate.  These tax codes along with any others you wish to use within Syrinx need to be set up via Syrinx SP Configuration\”Accounts/Invoicing”\Tax Codes.  You will then also need to set up nominal codes via Syrinx SP Configuration\”Accounts/Invoicing”\Nominal Codes to match the analysis codes as set up in Access Dimensions.  To view the sales analysis codes, go to Financials / Sales Ledger / Records / Analysis:

The values in the Analysis Code column are the nominal codes to be entered in Syrinx.  In a multi-depot scenario, you would have multiple analysis codes in Dimensions – one for each depot, such as:

  • RDG10110 – Hire Income, Reading depot
  • BRI10110 – Hire Income, Bristol depot

…and this would be entered as nominal code “|D|10110” in Syrinx.  The transfer program will then substitute the ‘|D|’ token for the ‘accounts department code’ that is entered against the depot records in Syrinx SP Configuration\General\Depots (i.e. ‘RDG’ or ‘BRI’).

If not populated, you will need to enter the following information via the “Enter New Connection Details” option prior to moving to the next step:

Password: Enter the password for the username. (Version dependent, sometimes “x”)

Considerations for Business Areas

If you have business areas enabled in Syrinx meaning that you are transferring information from one Syrinx database to multiple Sage company files, additional configuration is required.  You can verify if business areas are enabled in Syrinx SP Configuration\General\Depots.  If a “Bus Area” column is visible and populated with data, you are using business areas and further configuration of the tool is necessary:

 

 

To configure the transfer tool for use with business areas, open the transfer tool and click the “Edit” button in order to access the “Configuration Settings” screen.  The business area name needs to be entered exactly as it appears against the specific depot records.  You may need to open the depot record in the Syrinx Configuration program in order to see the full business area name in case it is cut off on the main “Depots” screen.  The business area name should be entered into the “Bus Area” field on the “Syrinx Data” tab:

 

You may also want to amend the “Name” value of this specific configuration entry to reflect the fact that it is linked to a business area:

 

After making that change, click “OK” to return to the main transfer tool screen and then click on the “Add” button in order to add an additional transfer tool configuration:  

You will then be prompted to add Syrinx database connection details and a log file location as per the initial installation of the transfer tool.  After entering the basic configuration information, set the appropriate Business Area name entry on the “Syrinx Data” tab and rename the entry so that it corresponds to this new business area configuration:



 

When opening the transfer tool, there will now be multiple configurations to choose from when running the transfer tool which correspond to the different business areas linked to Syrinx:

This process should be repeated for each Business Area that has been linked to Syrinx. 

Transfer Program Operation

Before running the transfer tool against an on-premise Dimensions database for the 1st time, the patch, CreateHCObjects.sql, found in the transfer tool’s installation directory (e.g. C:\Program Files (x86)\Higher Concept\SyrinxSP\SxAccess) should be applied to the Access Dimensions database.  This patch only needs to be applied once and would not need to be applied for any further Transfer tool installations.

To test the transfer, flag a few customers for export, then transfer them over.  The transfer program works by opening connections to both the Syrinx database and the Access Dimensions database, and then shuffling the data between the two.  To read/write the Access Dimensions database, the transfer program calls a set of views/procedures written and maintained by ourselves that in turn use a set of API views/procedures that form part of the core Access Dimensions product.  Our database objects are all prefixed “HC_” and they encapsulate all the logic of how we use their API.  

Note that the transfer program automatically posts the batches of invoices during the transfer – i.e. there is no need to go to Financials / Sales Ledger / Transactions / Post Batch after running the transfer program.  

Note also that the transfer program also re-calculates the aged debt figures automatically, so there is no need to go to System Control / Transaction Process / Aged Debt to do this manually after running the transfer program.
